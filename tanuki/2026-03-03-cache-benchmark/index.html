Tanuki: Cache Benchmark

3 marzo 2026 Server: tanuki.giobi.com (188.245.145.207) Hetzner cx33 · 4 siti WordPress
100/100

Cache implementata con successo

Tutti e 4 i siti rispondono in meno di 80ms. Il tempo di risposta medio è passato da ~560ms a ~64ms grazie a FastCGI Cache (full page cache a livello nginx) e Redis Object Cache (cache del database WordPress).

Misurato la velocità di partenza
Prima di toccare niente, abbiamo misurato quanto tempo ci metteva ogni sito a rispondere. Due siti (filippomarcodini.com e robertomotta.it) ci mettevano più di un secondo — troppo. Gli altri due erano già ragionevoli.
Attivata la cache delle pagine
Abbiamo detto al server: «quando qualcuno visita una pagina, salva la risposta. Se un altro visitatore chiede la stessa pagina, dagli la copia salvata invece di ricostruirla da zero». Questo velocizza enormemente le visite perché il server non deve interrogare il database ogni volta.
Attivata la cache del database
Anche internamente, WordPress ora salva in memoria le richieste al database più frequenti (impostazioni, menu, widget). Così anche quando la pagina non è in cache, il sito risponde più velocemente.
Aggiunta protezione contro i bot
I tentativi di accesso ripetuti (attacchi automatici) vengono ora limitati: massimo 3 tentativi al secondo sulla pagina di login. Inoltre, un punto di accesso tecnico spesso abusato (xmlrpc) è stato completamente bloccato.

Cache Stack implementato

1. FastCGI Cache (nginx-level full page cache) - Zone: WPCACHE, 100MB keys, 512MB max, 60m inactive - Path: /var/cache/nginx/fastcgi - Bypass: POST, logged-in users, wp-admin, wp-login, wp-json - Headers: X-FastCGI-Cache (HIT/MISS/BYPASS) 2. Redis Object Cache (WP database query cache) - Plugin: redis-cache (free, PhpRedis client) - Prefix per-site per evitare collisioni - Connessione: 127.0.0.1:6379 database 0 3. Rate Limiting (bot protection) - wp-login.php: 3 req/s burst 5 - xmlrpc.php: BLOCKED (deny all, return 403) - PHP in uploads/: BLOCKED 4. Static Assets - expires 365d, Cache-Control: public, immutable - access_log off per static files

Configurazione nginx (snippet)

# /etc/nginx/conf.d/fastcgi-cache.conf fastcgi_cache_path /var/cache/nginx/fastcgi levels=1:2 keys_zone=WPCACHE:100m max_size=512m inactive=60m use_temp_path=off; # /etc/nginx/snippets/fastcgi-cache.conf set $skip_cache 0; if ($request_method = POST) { set $skip_cache 1; } if ($http_cookie ~* "wordpress_logged_in|woocommerce_cart") { set $skip_cache 1; } fastcgi_cache WPCACHE; fastcgi_cache_valid 200 301 60m; fastcgi_cache_bypass $skip_cache; fastcgi_no_cache $skip_cache; add_header X-FastCGI-Cache $upstream_cache_status;

Note

- veterinariogravellona aveva Object Cache Pro (Cloudways) come drop-in con password Redis vecchia.
